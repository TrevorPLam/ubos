# Microsoft Style Guide\n\n**Official**: https://learn.microsoft.com/en-us/style-guide/  \n**Version**: 2.0 (2023)  \n**Purpose**: Clear, accessible, and consistent technical writing  \n**When to Use**: All documentation, user guides, and developer content\n\n---\n\n## üéØ Core Principles\n\n### 1. **Be Conversational**\n- Write like you're talking to a colleague\n- Use contractions (don't, it's, you'll)\n- Use \"you\" when addressing the reader\n- Avoid jargon; define it when necessary\n\n**‚ùå Avoid**: \"The aforementioned configuration parameter shall be modified\"\n**‚úÖ Use**: \"Change the configuration parameter\"\n\n---\n\n### 2. **Be Clear**\n- Simple words over complex ones\n- Short sentences over long ones\n- Active voice over passive voice\n- Specific examples over abstract concepts\n\n**‚ùå Avoid**: \"When configuration parameters have been established by the system administrator, the process can commence\"\n**‚úÖ Use**: \"After you set up the configuration, the process starts\"\n\n---\n\n### 3. **Be Concise**\n- Remove unnecessary words\n- One idea per sentence\n- One topic per paragraph\n- Use lists to break up text\n\n**‚ùå Avoid**: \"In the event that you should find yourself in a situation where the system is not responding appropriately to your inputs...\"\n**‚úÖ Use**: \"If the system doesn't respond...\"\n\n---\n\n### 4. **Be Consistent**\n- Use the same term for the same concept\n- Format similar items the same way\n- Use the same voice and tone\n- Follow style guide rules\n\n**‚ùå Avoid mixing**: \"user\", \"end-user\", \"person\", \"customer\"\n**‚úÖ Pick one**: \"user\" (unless distinguishing is essential)\n\n---\n\n### 5. **Be Helpful**\n- Start with what the reader wants to do\n- Anticipate questions\n- Provide examples\n- Link related content\n\n**‚ùå Avoid**: Dense technical reference\n**‚úÖ Use**: Task-oriented content with examples\n\n---\n\n## üìù Writing Standards\n\n### Audience\n\nWrite for **your least technical reader** without talking down to experts.\n\n**Questions to ask**:\n1. Who is reading this?\n2. What do they already know?\n3. What do they need to do?\n4. What might confuse them?\n\n---\n\n### Voice and Tone\n\n**Characteristics**:\n- **Knowledgeable**: Show expertise without arrogance\n- **Helpful**: Genuinely trying to help the reader\n- **Friendly**: Approachable, not stiff\n- **Professional**: Appropriate for business context\n\n**Examples - UBOS Context**:\n\n‚ùå **Stiff**: \"The system implements a multi-tenant architecture to ensure complete isolation between discrete customer datasets.\"\n\n‚úÖ **Friendly**: \"Each customer's data is completely separate from others, so they never see each other's information.\"\n\n---\n\n### Tense\n\n**Use**: Present tense (what the system does now)\n**Avoid**: Future tense when describing features\n**Exception**: When discussing planned features, use \"will\"\n\n```\n‚úÖ \"The API returns a JSON response\"\n‚ùå \"The API will return a JSON response\" (unless planned)\n```\n\n---\n\n### Person\n\n**Use**: Second person (\"you\") when instructing\n**Use**: First person plural (\"we\") when speaking as the company\n**Avoid**: First person singular (\"I\") in documentation\n\n```\n‚úÖ \"To create a new contact, click the + button\"\n‚ùå \"One should click the + button to create a new contact\"\n‚ùå \"Click the + button to create a new contact, I'm sure you'll enjoy it\"\n```\n\n---\n\n## üìã Formatting Standards\n\n### Headings\n\n**Rules**:\n- Use sentence case (first word capitalized)\n- Use descriptive, specific headings\n- Don't skip heading levels (no H1 ‚Üí H3)\n- Max three heading levels per page\n\n```markdown\n‚úÖ Creating a new tenant\n‚úÖ Setting up authentication\n\n‚ùå Creating A New Tenant (title case)\n‚ùå Understanding the System  (too vague)\n```\n\n### Lists\n\n**When to use**:\n- Three or more related items\n- Steps in a procedure\n- Options or choices\n\n**Format**:\n```markdown\nTo set up your account:\n1. Click Sign Up\n2. Enter your email\n3. Create a password\n4. Verify your email\n\nYou can choose:\n- Email notifications\n- SMS notifications\n- In-app notifications only\n```\n\n**Punctuation**:\n- Start each item with a capital letter\n- Use periods at the end if sentences are complete\n- If items are fragments: no period needed\n\n### Code Examples\n\n**Always include**:\n- What this code does\n- How to use it\n- Expected output\n\n```markdown\nTo authenticate a request, include your API key in the header:\n\n```javascript\nconst headers = {\n  'Authorization': 'Bearer YOUR_API_KEY'\n};\n```\n\nReplace `YOUR_API_KEY` with your actual key from your account settings.\n```\n\n### Tables\n\n**Use for**:\n- Comparing multiple items\n- Showing relationships\n- Reference information\n\n**Avoid using tables to**:\n- Create artificial structure\n- Force two-column layouts\n\n---\n\n## üî§ Word Choice\n\n### Common Word Preferences\n\n| Avoid | Use | Why |\n|-------|-----|-----|\n| Utilize | Use | Simpler, more direct |\n| Configure | Set up | More familiar |\n| Execute | Run | Clearer meaning |\n| Obtain | Get | More conversational |\n| Subsequently | Then | Shorter, clearer |\n| In the event that | If | Concise |\n| Per | According to | Clearer |\n| Shall | Should/Will | Modern English |\n| Aforementioned | [repeat the term] | Clearer |\n\n### Capitalization\n\n**Capitalize**:\n- Proper nouns (UBOS, PostgreSQL, Express)\n- Product names (UBOS Platform)\n- Feature names (Multi-Tenant Isolation)\n- Button names (Click the \"Save\" button)\n\n**Don't capitalize**:\n- General terms (database, API, server)\n- Roles (admin, user, operator)\n\n### Acronyms\n\n**First mention**: Spell out, then acronym\n```\n‚úÖ \"Multi-Tenant Isolation (MTI) ensures data separation. MTI is enabled by default.\"\n```\n\n**Common acronyms** (no need to spell out):\n- API\n- URL\n- HTTP/HTTPS\n- JSON\n- SQL\n\n---\n\n## ‚úÖ Content Checklist\n\n### Before Publishing\n- [ ] Title clearly describes the content\n- [ ] First paragraph states the purpose\n- [ ] Instructions use imperative mood (\"Click\", not \"You can click\")\n- [ ] No sentence is longer than 25 words\n- [ ] All code examples are tested\n- [ ] Links are descriptive (not \"click here\")\n- [ ] Spelling and grammar checked\n- [ ] Consistent terminology throughout\n\n### Quality Questions\n- [ ] Could a beginner understand this?\n- [ ] Would an expert find any errors?\n- [ ] Is it scannable (headings, lists, bold)?\n- [ ] Does it have a clear purpose?\n- [ ] Could it be shorter?\n\n---\n\n## üéØ UBOS Writing Examples\n\n### Getting Started (Conversational)\n```markdown\n# Getting Started with UBOS\n\nWelcome!
